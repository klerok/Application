# Frontend — Education Application

Визуальная часть приложения: страницы, формы и компоненты. Логику (вызов API, состояние авторизации, защита маршрутов) можно добавлять по плану ниже. Backend уже предоставляет API регистрации, входа и выхода.

---

## Что используется

- **React 18** — UI
- **Vite 6** — сборка и dev-сервер
- **React Router v7** — маршрутизация
- **CSS Modules** — стили в `styles/index.module.css` внутри папки каждого компонента/страницы/модуля; импорт: `import styles from './styles/index.module.css'`

---

## Как запустить проект

### Требования

- **Node.js** 18+ (рекомендуется 20+)
- **npm** (или yarn/pnpm)

### Установка зависимостей

Из корня репозитория:

```bash
cd frontend
npm install
```

### Режим разработки

```bash
npm run dev
```

Приложение откроется по адресу: **http://localhost:5173**

### Сборка для продакшена

```bash
npm run build
```

Собранные файлы появятся в папке `dist/`.

### Просмотр собранной версии

```bash
npm run preview
```

## Структура проекта

- **app** — корневой компонент приложения и маршруты.
- **core** — общая оболочка (layout, header) и глобальные стили.
- **pages** — страницы приложения; у каждой своя папка с `index.jsx`, `index.js`, `styles/index.module.css`.
- **modules** — переиспользуемые блоки логики/UI (например, форма авторизации), которые можно вставлять на страницы.
- **components** — переиспользуемые UI-компоненты (модалки, иконки и т.п.).
- Стили: **CSS Modules** в `styles/index.module.css` рядом с компонентом, использование `className={styles.имяКласса}`.

## Соответствие backend API

Backend (порт 3000) предоставляет следующие эндпоинты:

| Действие     | Метод | URL                     | Тело запроса                    | Ответ при успехе                    |
|-------------|-------|--------------------------|----------------------------------|-------------------------------------|
| Регистрация | POST  | `/api/auth/register`     | `{ username, email, password }`  | 200, `{ text: user }`               |
| Вход        | POST  | `/api/auth/login`        | `{ email, password }`            | 200, `{ token }` (JWT)              |
| Выход       | POST  | `/api/auth/logout`       | `{ token }`                      | 200, `{ message: "Logged out successfully" }` |

- При ошибке сервер возвращает **400** и `{ error }`.
- **Токен** после логина нужно сохранять (например, в `localStorage`) и при вызове logout передавать в теле запроса. Для будущих защищённых эндпоинтов — в заголовке `Authorization: Bearer <token>`.
- Базовый URL backend: `http://localhost:3000` (убедитесь, что backend запущен при проверке с фронта).

## План: что реализовать дальше

1. **API-слой**
   - Создать модуль (например, `src/core/api/auth.js` или в `modules/auth-form`) с функциями: `register(username, email, password)`, `login(email, password)`, `logout(token)`.
   - Использовать `fetch` с базовым URL `http://localhost:3000`.
   - В `logout` отправлять токен в теле запроса; после успешного ответа на клиенте удалять токен из хранилища и состояния.

2. **Состояние авторизации**
   - Выбрать способ: React Context, Zustand, Redux или подъём состояния (например, в App).
   - Хранить факт «пользователь залогинен» и при необходимости сам токен или данные пользователя (например, после логина сохранять токен в `localStorage` и в state).
   - При загрузке приложения проверять наличие токена в `localStorage` и при наличии считать пользователя залогиненным (опционально — проверка валидности через отдельный endpoint, если появится на backend).

3. **Обработка форм (Login, Register)**
   - В страницах `pages/Login` и `pages/Register`: управление полями через `useState`, сообщения об ошибках и загрузка (disabled при отправке).
   - В `onSubmit`: вызов API (`login` / `register`), при успехе — сохранение токена (для login), редирект (например, на главную или в личный кабинет), при ошибке — показ сообщения из ответа сервера.

4. **Защита маршрутов**
   - Реализовать обёртку (например, `ProtectedRoute`): если пользователь не авторизован — редирект на `/login` (или на главную).
   - Подключить к маршрутам, которые должны быть доступны только после входа.

5. **Header в зависимости от авторизации**
   - Не залогинен: ссылки «Вход» и «Регистрация».
   - Залогинен: отображать имя пользователя (если есть в state/context) и кнопку «Выйти»; по клику вызывать `logout(token)`, после успешного ответа удалять токен и сбрасывать состояние, редирект на главную или на `/login`.

6. **Выход (logout)**
   - На кнопке «Выйти»: отправить `POST /api/auth/logout` с телом `{ token }` (токен взять из `localStorage` или state).
   - После успешного ответа: удалить токен из `localStorage` и состояния, перенаправить пользователя.

7. **Работа с постами (когда понадобится)**
   - Backend имеет модель `Post` (title, content, authorId). Добавить модуль/страницы для списка постов и создания поста.
   - Запросы к API постов отправлять с заголовком `Authorization: Bearer <token>`.

8. **Обработка ошибок и загрузки**
   - Индикатор загрузки при отправке форм и при запросах к API.
   - Обработка сетевых ошибок и ответов 4xx/5xx, вывод понятных сообщений пользователю.

После выполнения шагов 1–6 у вас будет связанные фронт и бэк с регистрацией, входом и выходом; далее — защищённые маршруты и посты по плану.
